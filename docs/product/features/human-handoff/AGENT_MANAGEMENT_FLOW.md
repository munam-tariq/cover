## User Roles Explained

### Role: Owner (You)

**Who:** The person who created the project/account

**Can do:**
- Everything (full admin access)
- Invite/remove agents
- Configure handoff settings
- Manage knowledge base
- View analytics
- Access billing
- AND respond to chats (optional)

**Login:** Same as today (magic link to dashboard)

---

### Role: Agent

**Who:** Team members invited by owner

**Can do:**
- View agent dashboard (inbox, queue)
- Respond to customer chats
- Set their status (online/away/offline)
- See other team members' status
- Resolve conversations

**Cannot do:**
- Change project settings
- Invite other agents
- Access knowledge base
- View analytics
- Access billing

**Login:** Same magic link flow, but sees Agent Dashboard instead of Admin Dashboard

## How the System Knows Who Sees What

```typescript
// When user logs in, check their role for this project
async function getUserDashboard(userId: string, projectId: string) {
  // Check if user is the project owner
  const project = await db.projects.findOne({ id: projectId });

  if (project.user_id === userId) {
    return 'owner_dashboard';  // Full admin access
  }

  // Check if user is an agent for this project
  const membership = await db.project_members.findOne({
    user_id: userId,
    project_id: projectId,
    role: 'agent'
  });

  if (membership) {
    return 'agent_dashboard';  // Limited agent access
  }

  return 'no_access';  // Not authorized
}
```

## Database Changes Needed

### New Table: `project_members`

```sql
CREATE TABLE project_members (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,  -- NULL if pending

  -- Invitation
  email TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'agent' CHECK (role IN ('agent', 'admin')),
  status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'active', 'removed')),

  -- Invitation tracking
  invited_by UUID NOT NULL REFERENCES auth.users(id),
  invited_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  accepted_at TIMESTAMPTZ,

  -- Settings
  max_concurrent_chats INTEGER NOT NULL DEFAULT 5,

  -- Timestamps
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  -- Constraints
  UNIQUE(project_id, email)
);
```

### Invitation Flow in Database

```
1. Owner invites sarah@company.com
   â†’ INSERT project_members (email, status='pending', user_id=NULL)

2. Sarah clicks invite link, creates account
   â†’ UPDATE project_members SET user_id=xxx, status='active', accepted_at=NOW()

3. System creates agent_availability record
   â†’ INSERT agent_availability (user_id, project_id, status='offline')

4. Sarah sets status to Online
   â†’ UPDATE agent_availability SET status='online'
```

## FAQ

### Q: Can an agent be on multiple projects?
**A:** Yes! If Agent Sarah is invited to Project A and Project B, she can switch between them. Each project has separate chat limits and status.

### Q: What if the owner wants to respond to chats too?
**A:** Owners automatically have agent capabilities. They can set their status to Online and respond to chats just like agents.

### Q: What happens when an agent is removed?
**A:** Their active chats are moved back to the queue. They lose access immediately.

### Q: Can agents see each other's chats?
**A:** Agents can see the queue (waiting chats) and their own chats. They cannot see other agents' active conversations.

### Q: What if no agents are online?
**A:** Chats stay in queue. If business hours are enabled and it's outside hours, customers see the offline form.
